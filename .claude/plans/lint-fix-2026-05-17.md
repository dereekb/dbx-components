### Bucket C — `require-single-return` refactors (51 warnings)

**Strategy:** Manual — no autofix. Apply `dbx__note__typescript-programming` → Single Return Per Function. Each function should accumulate the result in a `let` and have a single `return` at the bottom (or hoist branching into a helper).

**Project priority:**
- `dbx-form` (33) — concentrated in formly/forge files. Read the surrounding code; many are likely small early-return patterns that collapse cleanly.
- `util` (16)
- `dbx-web` (2)

Common pattern:

```ts
// Before
function f(x: T): U {
  if (cond) return a;
  return b;
}

// After
function f(x: T): U {
  let result: U = b;
  if (cond) result = a;
  return result;
}
```

For multi-branch logic, sometimes the cleanest fix is to extract an inner helper, return its result, and keep the outer function single-return. Don't pile on `let` reassignments where a helper is clearer.

## Conventions for all buckets

- Use the existing surrounding code as the source of truth. If a function's purpose is unclear from its body and call sites, mark it for follow-up rather than guessing.
- Follow `dbx__note__typescript-jsdocs` for any JSDoc work.
- Follow `dbx__note__typescript-programming` for code-shape rules (single return, readonly, Maybe<T>, factory naming).
- Don't pile unrelated changes into a lint-fix commit — keep diffs reviewable.
- Don't run `--fix` on `*.spec.ts` files at the same time as source files; the spec config disables several rules and the fixer can churn. Run source files first, then specs separately if needed.
