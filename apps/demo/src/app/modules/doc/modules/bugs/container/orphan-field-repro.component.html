The production "Create Another Job" flow on HelloSubs' school create-job page emits 9–11 NG01902 / Orphan field errors per reset cycle. The errors are thrown from node_modules/@angular/forms/fesm2022/_validation_errors-chunk.mjs:955 whenever Signal Forms' children-map sees a registered slot drop out of the parent value.

The mechanism: a dbx-forge form runs with dbxFormSourceMode="always" , which re-applies the source observable on every emission. The source is a reset / per-field setValue cascade whose partial emissions drop the dateScheduleRange slot — Angular treats the orphaned field as an error and pays a stack-formatting cost that freezes Chrome on slower hardware.

ng-forge's DynamicForm layer cannot reproduce the orphan in isolation: FormStateManager.entity deep-merges declared defaults into the partial input. The bug lives above that layer, in @dereekb/dbx-form 's DbxFormSourceDirective + DbxForgeFormComponent plumbing. This page is the simplest harness that wires those two together with the calendar field and lets you drive the cascade by hand.
